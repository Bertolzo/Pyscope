# Branching Strategy

This document describes the branching strategy for PyScope. The goal is to
keep `main` as the single source of truth while enabling feature/experiment
branches and safe integration via `develop`.

Principles
- `main`: stable baseline, always CI-green and reviewed. Public-facing.
- `develop`: integration branch for features and experiments; merges to `main`
  happen via reviewed PRs.
- `scope/*`: short-lived experiment branches (visualizer, atlas, etc.).
- `fix/*`: urgent patches and hotfixes.
- Keep branches small, focused and short-lived.

Suggested workflow
1. Create a feature branch from `develop`: `git checkout -b scope/visual-004 develop`.
2. Implement and add tests. Run local tests and CI.
3. Open a Pull Request into `develop` for integration and review.
4. Merge `develop` into `main` via a reviewed PR when ready for release.

Branch protection recommendations (GitHub)
- Protect `main`: require PR reviews, require CI checks, disallow direct pushes.
- Require status checks for `develop` when merging to `main`.

Housekeeping
- Delete obsolete `scope/*` branches from remote after merging or if abandoned.
- Keep `scope/*` names descriptive: `scope/visual-<n>`, `scope/atlas-<n>`.

This is a lightweight, iterative model intended to keep `main` central and
clean while enabling rapid experimentation in `scope/*` branches.
